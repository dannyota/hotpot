# Documentation Style

Write clear, scannable documentation. **Keep it short.**

## ‚ö° Golden Rule

**Nobody wants to read long documentation.** Get to the point.

- One sentence > three sentences
- Table > paragraph
- List > essay
- Delete unnecessary words
- If a section feels boring, it probably is ‚Äî cut it

## ‚úçÔ∏è Writing

### Voice

- **Active** - "Ent generates" not "is generated by"
- **Present** - "stores" not "will store"
- **Direct** - "Use edges" not "You should use edges"
- **No fluff** - Delete "simply", "just", "obviously"

### Structure

Start every doc with one clear sentence:

```markdown
# Feature Name

What this document covers in one sentence.
```

**Good:**
- "Temporal activities for data ingestion and processing."
- "Type-safe ent schema patterns for bronze and history tables."

**Bad:**
- "This document will explain how temporal activities work..."
- Long paragraphs before getting to the point

## ‚ö†Ô∏è Avoid

| Don't | Do |
|-------|-----|
| Long paragraphs | Lists or tables |
| "Information About X" | "X" (just the topic) |
| Future tense | Present tense |
| Passive voice | Active voice |
| Raw URLs | Descriptive links |
| "Simply" / "just" | Delete these words |
| Explaining obvious things | Focus on non-obvious |
| Walls of text | Scannable structure |

### Bad Example

```markdown
## Information About Configuration

In this section, we will explain how configuration works.
The configuration is loaded by the system and it will be
used to determine how the application behaves...
```

### Good Example

